# AI Smeta KZ — архитектура

Приложение готовит **базовую (ориентировочную) смету здания** по строительным
нормам Республики Казахстан. Заказчик вводит понятные параметры объекта, система
рассчитывает объёмы материалов и работ и формирует смету с разбивкой по разделам,
накладным, резерву и НДС.

Ключевая идея экономии токенов: **нормативные знания (ГОСТ/СНиП/СН РК) один раз
разбираются LLM и складываются в базу данных как структурированные правила**.
Повторные расчёты того же профиля объекта берут готовые правила из БД и не
обращаются к LLM.

## Поток (роли и системы)

```
┌──────────┐  1. атрибуты   ┌──────────┐  3a. сигнатура входа   ┌──────────────┐
│ Frontend │ ─────────────► │ Backend  │ ─────────────────────► │ knowledge_   │
│ (HTML/JS)│                │ FastAPI  │ ◄───── hit ─────────── │ cache (БД)   │
│          │ ◄─5. статус──  │          │                        └──────────────┘
│          │   (SSE)        │          │  3b. miss → реестр норм      │
│          │                │          │ ─────────────────────► ┌──────────────┐
│          │                │          │  извлечение через LLM   │ norm_docs /  │
│          │                │          │  (системный промпт РК)  │ norm_rules   │
│          │  6. смета JSON │          │ ◄── правила + кэш ───── └──────────────┘
│          │ ◄───────────── │          │  4. расчёт объёмов + цены → смета
└──────────┘  7. рендер     └──────────┘
```

1. Пользователь вводит атрибуты здания (тип, габариты, этажность, конструктив,
   фундамент, класс отделки/инженерии, проценты накладных/резерва/НДС, виды работ).
2. Фронт `POST /api/estimate` → бэк создаёт **job** и сразу возвращает `job_id`.
3. Бэк в фоне:
   - **3a. Разбор атрибутов** → нормализация в `BuildingInput`, вычисление
     `signature` (хэш профиля, влияющего на нормы).
   - **3b. Резолв норм** (`norms/resolver.py`):
     - `knowledge_cache` по сигнатуре → **hit**: берём готовый `NormProfile`.
     - **miss**: из `norm_documents` (засеяны из MD) берём профильные документы
       по типу объекта; для каждой нужной категории правил ищем в `norm_rules`;
       чего нет — **извлекаем через LLM** (системный промпт из MD), валидируем,
       пишем в `norm_rules`, собираем `NormProfile`, кладём в `knowledge_cache`.
4. **Расчёт** (`calc/`): по геометрии и нормативным коэффициентам считаем объёмы
   (бетон, арматура, опалубка, кладка, тепло/гидроизоляция, кровля, фасад,
   остекление, отделка, инженерные сети, земляные работы), применяем цены
   (`calc/pricing.py`) → строки сметы → накладные/резерв/НДС → итог.
5. Всё это время фронт слушает `GET /api/estimate/{job_id}/events` (SSE) и
   показывает шаги: «Разбор атрибутов» → «Поиск норм в БД» → «Извлечение норм
   (СН РК …)» → «Расчёт объёмов» → «Формирование сметы» → «Готово».
6. Бэк возвращает финальный JSON: класс точности, нормативные источники,
   расчёт объёмов с формулами, таблица сметы, итоги, вопросы подрядчику.
7. Фронт рендерит смету, даёт экспорт TXT (как в примере) и копирование.

## Модель данных (knowledge base)

- **norm_documents** — реестр нормативных документов (код, название, тип
  `СН РК/СНиП/СП РК/ГОСТ/ТР`, url, применимые типы объектов, статус
  `seed/parsed/stub`, выдержка, дата загрузки). Засеян из `akty_i_trebovaniya…md`.
- **norm_rules** — извлечённые **структурированные правила**: `category`
  (`concrete_consumption`, `rebar_consumption`, `formwork_ratio`,
  `partition_ratio`, `wall_insulation`, `foundation`, `glazing_ratio`, …),
  `value`, `unit`, `conditions` (JSON: `{structure_type, foundation_type, …}`),
  `confidence`, `source` (`seed/llm/document`), `note`, FK на документ.
  **Сердце системы — расчётный движок читает именно их.**
- **knowledge_cache** — собранный `NormProfile` под сигнатуру входа (с TTL),
  чтобы повторные запросы не дёргали LLM.
- **price_items** — справочник цен (ресурс, ед., цена материала/работы/машин,
  регион, источник). Засеян из примера сметы.
- **estimates** — сохранённые расчёты (вход, результат, итоги, дата).
- **jobs** — трекинг асинхронных задач для статусов (статус, прогресс, шаги,
  ссылка на результат).

## Слои бэкенда

```
app/
  config.py        — настройки (env), пути
  database.py      — engine, session, Base
  models.py        — ORM (см. выше)
  schemas.py       — Pydantic: BuildingInput, NormProfile, Estimate, Job…
  llm/             — провайдер-абстракция: base + gemini/anthropic/openai/demo
  norms/
    registry.py    — сид документов из MD
    defaults.py    — дефолтные коэффициенты (fallback, всегда дают результат)
    extractor.py   — извлечение правил LLM по системному промпту РК
    resolver.py    — резолв профиля норм (cache → rules → LLM)
  calc/
    geometry.py    — производные геометрии (площадь застройки, объём, периметр)
    volumes.py     — объёмы по разделам из геометрии + норм
    pricing.py     — справочник цен + резолв цены ресурса
    estimate.py    — сборка строк сметы, итоги, накладные/резерв/НДС
  jobs/manager.py  — реестр job'ов + публикация событий (SSE)
  api/routes.py    — REST + SSE
  seed.py          — инициализация БД сидами
  main.py          — сборка приложения, отдача фронта
```

## Принципы

- **Детерминированный расчёт.** LLM влияет только на нормативные коэффициенты;
  сама арифметика прозрачна, проверяема и приводит формулы (как в примере сметы).
- **Всегда есть результат.** Если LLM недоступен/не уверен — берутся
  `defaults.py`, строка помечается «требует проверки сметчиком» (не выдумываем
  номера норм — отражено в системном промпте).
- **Провайдеро-независимость.** Один интерфейс `LLMProvider`; ключ и модель из env.
- **Кэш во главе угла.** Сигнатура входа → готовый профиль норм; экономия токенов.
