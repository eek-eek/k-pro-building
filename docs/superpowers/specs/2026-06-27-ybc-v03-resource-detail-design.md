# Ресурсная детализация сметы (v0.3) — дизайн

**Дата:** 2026-06-27
**Статус:** утверждён к планированию
**Предыдущее:** [v0.2](2026-06-27-yale-building-calculator-v02-design.md) (версии, ИИ-чат, настройки, редизайн, рекомендации)

## Цель

Превратить смету из укрупнённой (класс 5) в **ресурсную (класс 3)**: каждая
работа раскладывается на ресурсы — **материалы, труд, машины** — с нормами
расхода на единицу работы. Ресурсы **редактируемы**: правка расхода или цены
ресурса пересчитывает строку и итоги снизу вверх. Детализируются **все 13
разделов** сметы.

### Класс точности и честность

Нормы расхода и цены ресурсов **курируются нами** как ориентировочные показатели
РК (инженерная практика), а **не** взяты из лицензированных сборников ГЭСН/СН РК.
Каждая ресурсная строка наследует пометку «ориентировочно, требует проверки
сметчиком». Смета остаётся инструментом предварительной оценки; добавление
ресурсного уровня повышает прозрачность расчёта, но не делает её основанием для
договоров подряда. Существующие предупреждения сметы сохраняются.

## Архитектурное решение

### Ключевое упрощение: без новых таблиц БД

Ресурсы **снимаются снимком на строку сметы** (так же, как сейчас на строку
снимаются цены работ), а каталог норм живёт в коде. Это исключает миграции и риск
для существующих смет/версий.

- Каталог норм — новый код-модуль `backend/app/calc/resource_catalog.py`.
- При расчёте `build_estimate` кладёт в каждую строку снимок ресурсов и выставляет
  цены строки = свёртка состава.
- Версии сериализуют ресурсы вместе со строкой через существующий `to_jsonable`
  — изменений в `models.py` нет.
- Глобальное редактирование каталога в Настройках — **вне области** этой версии
  (правка ресурсов идёт на уровне конкретной сметы через снимок).

### Авторитет расчёта — ресурсы

Цена строки **выводится** из ресурсов, а не задаётся напрямую:

```
material_price = Σ (consumption × price) по ресурсам kind="material"
labor_price    = Σ (consumption × price) по ресурсам kind="labor"
machine_price  = Σ (consumption × price) по ресурсам kind="machine"
total          = round(quantity × (material_price + labor_price + machine_price))
```

`consumption` — норма расхода ресурса на **единицу работы**, поэтому
`material_price` и т.д. сохраняют прежнюю семантику «за единицу работы». **Формула
итога строки не меняется** — значит золотой инвариант, разделы, накладные/резерв/
НДС, рекомендации, откат остаются совместимыми.

## Модель данных

### Новая схема `ResourceLine` (`schemas.py`)

```python
class ResourceLine(BaseModel):
    code: str            # код ресурса, уникальный в пределах работы
    name: str            # «Бетон товарный B25», «Бетонщик 4 р.», «Кран башенный»
    kind: str            # "material" | "labor" | "machine"
    unit: str            # «м³», «чел-ч», «маш-ч», «т», «м²», «шт», «кг»
    consumption: float   # норма расхода на единицу работы (редактируется)
    price: float = 0.0   # цена ресурса за его единицу (редактируется)
```

Производные величины (не хранятся, считаются на отображении из `consumption`,
`price` и `quantity` родительской строки):
- объём ресурса в смете = `consumption × line.quantity`;
- сумма ресурса = `round(consumption × line.quantity × price)`.

### Расширение `EstimateLine` (`schemas.py`)

```python
class EstimateLine(BaseModel):
    ...                              # без изменений
    resources: list[ResourceLine] = []   # пусто = строка без детализации
```

Пустой список `resources` = строка работает как сейчас (плоская цена). Это
относится к подготовительным работам (1.1), рекомендациям и любой работе без
состава в каталоге.

## Каталог ресурсов (`calc/resource_catalog.py`)

Структура: `work_key → list[ResourceSpec]`, где `work_key` — ключ объёмной
позиции из `volumes.py` (`excavation`, `foundation_concrete`, `frame_concrete`,
`rebar`, `formwork`, `partitions`, `roof`, `facade`, `glazing`, `screed`,
`wall_finish`, `ceiling_finish`, `hvac`, `plumbing`, `electrical`, `low_current`,
`landscaping`, `waterproofing_foundation`, `insulation_*`, `soil_removal`,
`backfill`).

```python
@dataclass(frozen=True)
class ResourceSpec:
    code: str
    name: str
    kind: str        # "material" | "labor" | "machine"
    unit: str
    consumption: float
    price: float

COMPOSITIONS: dict[str, list[ResourceSpec]] = {
    "foundation_concrete": [   # на 1 м³ бетона фундамента
        ResourceSpec("concrete_b25", "Бетон товарный B25", "material", "м³", 1.02, 30000),
        ResourceSpec("pump",        "Бетононасос",         "machine",  "маш-ч", 0.12, 18000),
        ResourceSpec("vibrator",    "Вибратор глубинный",  "machine",  "маш-ч", 0.15, 1500),
        ResourceSpec("concreter",   "Бетонщик 4 р.",       "labor",    "чел-ч", 1.8,  3500),
    ],
    "excavation": [            # на 1 м³ разработки грунта
        ResourceSpec("excavator", "Экскаватор 1 м³",   "machine", "маш-ч", 0.03, 22000),
        ResourceSpec("operator",  "Машинист экскаватора", "labor", "чел-ч", 0.03, 4000),
    ],
    # … все 13 разделов
}
```

**Соглашения:**
- единицы труда — `чел-ч`, машин — `маш-ч`; цены ресурсов — индикативные РК
  (Астана, 2026), согласованы по порядку с текущим `PRICES`;
- свёртка состава работы должна давать цену **того же порядка**, что текущая
  плоская цена в `PRICES` (sanity-проверка при курировании), чтобы итоги смет не
  «прыгнули» при переходе на ресурсный расчёт;
- каждая объёмная `work_key` из всех 13 разделов имеет непустой состав
  (проверяется тестом полноты).

### Хелпер свёртки (`calc/resource_catalog.py`)

```python
def rollup(resources: list[ResourceLine]) -> tuple[float, float, float]:
    """(материал, труд, машины) за единицу работы = Σ consumption×price по виду."""
    m = sum(r.consumption * r.price for r in resources if r.kind == "material")
    l = sum(r.consumption * r.price for r in resources if r.kind == "labor")
    k = sum(r.consumption * r.price for r in resources if r.kind == "machine")
    return m, l, k

def snapshot_for(work_key: str) -> list[ResourceLine]:
    """Свежий снимок ResourceLine для работы (пустой, если состава нет)."""
```

`rollup` — **единственная** точка свёртки, используется и в `build_estimate`, и в
`recompute_estimate` (гарантия совместимости).

## Движок

### `build_estimate` (`calc/estimate.py`)

В цикле по разделам, после вычисления объёма:

```python
resources = snapshot_for(key)
if resources:
    material, labor, machine = rollup(resources)
else:
    price = get_price(db, key, region)          # как сейчас
    material, labor, machine = price.material, price.labor, price.machine
line = EstimateLine(..., material_price=material, labor_price=labor,
                    machine_price=machine, resources=resources,
                    total=round(quantity * (material + labor + machine)))
```

Подготовительная строка (1.1) и рекомендации — без ресурсов (как сейчас).

### `recompute_estimate` (`calc/estimate.py`)

Для каждой основной строки:

```python
if ln.resources:
    ln.material_price, ln.labor_price, ln.machine_price = rollup(ln.resources)
unit_cost = ln.material_price + ln.labor_price + ln.machine_price
ln.total = round(ln.quantity * unit_cost)
```

Так правка ресурсов (расход/цена) даёт новые цены строки → новый итог. Строки без
ресурсов считаются по своим ценам, как сейчас. Перенос контекста (warnings,
sources, volumes, prep, разделы, итоги) — без изменений.

### Контракт-инвариант

`build_estimate` выставляет `material_price = rollup(resources)`; `recompute`
выводит то же из тех же ресурсов. Значит `recompute` на нетронутой смете —
**no-op** (существующий тест `test_recompute_is_noop_on_unedited_estimate`
остаётся зелёным). Это центральный тест ресурсного уровня.

## API

**Новых эндпоинтов нет.** Правка ресурсов идёт через существующий
`POST /estimates/{id}/manual-edit`, который принимает полные строки (теперь с
`resources`), вызывает `recompute_estimate` и создаёт версию `manual_edit`.
Сериализация ресурсов — через текущий `to_jsonable`. Чат-правки, рекомендации,
откат, версии — без изменений кода.

## Фронтенд (`app.js`, `styles.css`)

### Таблица сметы — раскрытие и редактирование ресурсов

- Строка работы **с ресурсами** получает кнопку-раскрытие (▸/▾). При раскрытии под
  строкой показываются ресурсные подстроки, сгруппированные по виду
  (материалы / труд / машины): `название · ед. · расход (input) · объём (произв.)
  · цена ресурса (input) · сумма (произв.)`.
- Для строки **с ресурсами** ячейки цены работы (материал/работа/машины) —
  **только чтение** (выводятся из ресурсов); редактируется `объём` работы и поля
  ресурсов (`расход`, `цена`). Изменение объёма работы масштабирует объёмы
  ресурсов.
- **Полное редактирование состава:** в раскрытой строке можно **добавить ресурс**
  (кнопка «+ ресурс»: выбор вида material/labor/machine, ввод названия, единицы,
  расхода, цены) и **удалить ресурс** (кнопка в строке ресурса). Цены работы
  пересчитываются из нового набора.
- Строка **без ресурсов** — редактируется как сейчас (ячейки цены вводимы);
  «+ ресурс» доступна и тут (превращает плоскую строку в ресурсную).
- `saveManualEdit` собирает: объём работы (и цены для строк без ресурсов), для
  ресурсных строк — полный массив ресурсов (`code, name, kind, unit, consumption,
  price`); шлёт строки (с `resources`) в `manual-edit`. Существующий поток
  сохранения/версии сохраняется. Движок уже считает из любого набора ресурсов.

### Экспорт в Word (`exportDocx`)

Под каждой работой выводятся её ресурсы вложенными строками (с отступом),
сгруппированные по виду, с расходом/объёмом/ценой/суммой. Работы без ресурсов —
как сейчас.

## Источник цен материалов (адаптер + Satu-провайдер)

Цены ресурсов по умолчанию — курируемые. Внешние источники подключаются за
**единым адаптером**, цены **материалов** можно опционально обновлять из них.
Труд и машины — всегда курируемые (на маркетплейсах их нет).

### Интерфейс (`backend/app/pricesource/`)

```python
@dataclass
class PriceQuote:
    price: float
    source: str          # "curated" | "satu"
    fetched_at: str      # ISO-дата
    note: str = ""       # напр. «медиана 12 предложений, розница»

class PriceSource(Protocol):
    name: str
    def quote_materials(self, codes: list[str]) -> dict[str, PriceQuote]: ...
```

- **`CuratedSource`** — дефолт, отдаёт цены из `resource_catalog` (поведение как
  сейчас).
- **`SatuSource`** — для каждого материала по **конфигу сопоставления**
  (`resource_code → URL/запрос категории Satu`) тянет страницу категории,
  извлекает цены из HTML, берёт **медиану** валидных предложений, нормирует к
  единице ресурса. Только материалы.

### Поведение SatuSource (оговорки заложены в дизайн)

- **Опционально и по запросу** — не при каждом расчёте; вызывается явно
  (обновление цен материалов сметы).
- **Кэш с TTL** — повторные запросы из кэша; низкая частота обращений к Satu
  (антибот/ToS).
- **Совещательно** — возвращает **предлагаемые** цены с пометкой источник+дата+
  заметка («розница»); сметчик применяет к ресурсам (затем recompute). Не
  применяется молча.
- **Мягкая деградация** — при сети/блокировке/пустом парсе для кода → откат на
  `CuratedSource`; расчёт **никогда не падает**.
- Парсер изолирован; падение разметки Satu = пустой результат для кода, не
  исключение.

### API

`POST /estimates/{id}/suggest-material-prices` с `{source: "satu"}` → для
материальных ресурсов сметы возвращает `{resource_code: PriceQuote}`
(предложения для ревью). Применение — обычная правка ресурсов через `manual-edit`.
`GET /price-sources` → список доступных провайдеров.

## Обратная совместимость

- Старые версии смет (в БД без `resources`) десериализуются с `resources = []`
  (значение по умолчанию) → ведут себя как плоские строки. Никаких миграций.
- `models.py`, эндпоинты, чат, рекомендации, откат — без изменений сигнатур.
- Разделы, накладные/резерв/НДС, золотой recompute — формула не тронута.

## Тестирование

1. `rollup`: сумма по видам корректна; пустой список → нули.
2. Инвариант: `build → recompute` без правок — строки и итоги идентичны (с
   ресурсами).
3. Правка расхода ресурса → меняется `material/labor/machine_price` строки и
   `grand_total` в ожидаемую сторону.
4. Правка цены ресурса → аналогично.
5. Изменение объёма работы масштабирует суммы ресурсов (объём ресурса =
   расход×объём работы), итог строки корректен.
6. Строка без ресурсов (prep, рекомендация) — поведение как сейчас.
7. Round-trip сериализации версии с ресурсами (`to_jsonable` → JSON → `EstimateResult`).
8. Полнота каталога: каждая объёмная `work_key` всех 13 разделов имеет непустой
   состав; `rollup` каждой даёт цену того же порядка, что текущий `PRICES`
   (допуск, напр. ±40%).
9. API: `manual-edit` с изменёнными ресурсами создаёт версию и пересчитывает.
10. Добавление/удаление ресурса в строке → пересчёт цены работы и итога; удаление
    всех ресурсов → строка считается по оставшимся ценам (или нулю).
11. `CuratedSource.quote_materials` возвращает курируемые цены.
12. `SatuSource`: парсер извлекает цены из фикстуры HTML и берёт медиану; при
    пустом/битом HTML для кода → откат на курируемую цену (мок, без реальной сети);
    труд/машины не запрашиваются.

## Область и раскатка (для планов)

Реализуется как **v0.3** несколькими планами:

1. **План 1 — схема + движок + recompute + тесты** на 1–2 разделах (земляные,
   фундаменты/монолит): `ResourceLine`, `EstimateLine.resources`,
   `resource_catalog.py` (формат + `rollup` + `snapshot_for` + первые составы),
   интеграция в `build_estimate`/`recompute_estimate`, тесты 1–7, 9.
2. **План 2 — наполнение каталога** по всем 13 разделам (курирование составов),
   тест полноты (8).
3. **План 3 — фронтенд**: раскрытие строк, редактируемые ресурсы (правка +
   добавление/удаление), сбор правок, Word-экспорт с вложенными ресурсами.
4. **План 4 — источник цен**: адаптер `pricesource`, `CuratedSource`,
   `SatuSource` (парсер HTML + медиана + кэш + мягкая деградация), эндпоинт
   `suggest-material-prices`, UI-кнопка «обновить цены материалов из Satu» с
   ревью предложений. Тесты на фикстурах, без реальной сети в тестах.

Каждый план оставляет приложение в рабочем состоянии (разделы без состава
продолжают считаться по плоской цене до их наполнения; источник цен опционален).

## Вне области (следующие версии)

- Глобальное редактирование каталога ресурсов в Настройках (сейчас — правка
  снимка на уровне сметы).
- Импорт реальной базы ГЭСН/СН РК/РДС из файлов; официальные сборники сметных
  цен РК как провайдер источника.
- Хранение ресурсов и каталога в отдельных таблицах БД.
- LLM-формирование ресурсного состава.
- **OLX-провайдер** — доска объявлений, шумные/розничные/частные цены; как
  источник для сметы слабее Satu, отложен.
- Покупательское API Satu для чтения чужих цен — **не существует** (публичное API
  Satu предназначено продавцам для своего магазина); поэтому только HTML-парсинг.
