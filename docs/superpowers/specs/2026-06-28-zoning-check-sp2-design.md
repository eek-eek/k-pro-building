# SP2 — Проверка участка по генплану/кадастру — дизайн

**Дата:** 2026-06-28
**Статус:** утверждён к планированию (развилки согласованы с пользователем)
**Связь:** расширяет SP1 (объекты + карта). Добавляет проверку зоны к карточке объекта.

## Видение
На карточке объекта — кнопка **«Проверить участок»**. По нажатию система запрашивает
национальный геопортал по координате объекта и возвращает вердикт: **разрешено / ограничено /
не проверено** — с деталями (кадастровый номер, целевое назначение участка) и
предупреждением, если назначение участка не вяжется с типом строящегося объекта. На карте —
переключаемый слой кадастра/водоохранных зон поверх спутника.

## Честная рамка (по результатам спайка осуществимости 2026-06-28)
Прямого публичного слоя «функциональное зонирование генплана» через машиночитаемый API **нет**.
Но национальный геопортал отдаёт через WFS то, что отвечает на тот же вопрос «годится ли участок
и подо что он размечен»:

- **Источник:** GeoServer на `https://map.gov.kz/geoserver/ows` (тот же сервер дублируется на
  `ggk.kz/map_gov_kz/geoserver`). **WFS 2.0**, `outputFormat=application/json`, CRS `EPSG:4326`.
  Покрытие **национальное** — Алматы и Астана (портал Астаны genplan.saulet.astana.kz из РФ/КЗ был
  недоступен в спайке, но национальный слой делает его не нужным).
- **Слой `openmap:land_plots`** — кадастровые участки. Атрибуты (проверены живьём):
  `kad_nomer` (кад. номер), `tsn_ru` (**целевое назначение** = разрешённое использование),
  `squ` (площадь, м²). Геометрия — MultiPolygon в EPSG:4326. Это де-факто «зонирование» на
  уровне участка.
- **Слой `geonode:almaty_waterprotectionzone`** (и `*band`) — водоохранные зоны = жёсткое
  ограничение застройки. Существуют для ряда регионов (almaty, karagandy, zht, …); для Астаны
  отдельного слоя в каталоге может не быть — проверка водоохранных зон best-effort по доступным
  региональным слоям.

**Вывод спайка:** автоматическая проверка по координате **реализуема** (WFS live, GeoJSON,
точечный пространственный запрос отрабатывает). Поэтому строим `WfsZoningProvider` с мягкой
деградацией на «не проверено», а НЕ только ручное наложение.

## Механика запроса (валидирована в спайке)
Пространственный фильтр — параметром `bbox` вокруг точки (не требует имени геометрии в CQL):
```
GET https://map.gov.kz/geoserver/ows
  ?service=WFS&version=2.0.0&request=GetFeature
  &typeNames=openmap:land_plots
  &outputFormat=application/json&srsName=EPSG:4326
  &count=5
  &bbox=<lat-d>,<lon-d>,<lat+d>,<lon+d>,urn:ogc:def:crs:EPSG::4326
```
Ось для urn `EPSG::4326` в WFS 2.0 — **lat,lon**. `d` — крошечная дельта (≈0.00015°, ~15 м), чтобы
гарантированно зацепить участок под точкой. Из вернувшихся участков берём тот, что реально
содержит точку (point-in-polygon на стороне сервиса через малый bbox; при нескольких — ближайший
по центроиду / первый). Площадь/кад.номер/назначение — из его свойств.

Те же параметры с `typeNames=geonode:<region>_waterprotectionzone` → попадание в водоохранную зону.

## Модель данных
Расширяем существующую таблицу `building_objects` (миграция — идемпотентный ALTER-guard, как
`object_id` в SP1; БД dev уже содержит таблицу без этих колонок):
```
zone_status      str | None   # allowed | restricted | unknown (None — ещё не проверяли)
zone_land_use    str = ""     # tsn_ru — целевое назначение участка
zone_kad_nomer   str = ""     # кадастровый номер
zone_note        str = ""     # пояснение/предупреждение о несоответствии
zone_checked_at  datetime|None
```
Вердикт кэшируется на объекте (повторный заход не дёргает портал). Кнопка «Проверить» делает
свежий запрос и перезаписывает эти поля.

## Схемы (`schemas.py`)
```python
class ZoneVerdict(BaseModel):
    status: str                      # allowed | restricted | unknown
    land_use: str = ""               # целевое назначение (tsn_ru)
    kad_nomer: str = ""
    zone: str = ""                   # напр. "водоохранная зона" | "кадастровый участок" | ""
    source: str = "map.gov.kz/geoserver (WFS)"
    note: str = ""                   # предупреждение/пояснение
    checked_at: Optional[str] = None
```

## Бэкенд: адаптер зонирования (`app/zoning/`)
Изолированный модуль с провайдером (по образцу `pricesource/`):
- `app/zoning/base.py` — `ZoneVerdict`-логика-независимый контракт `ZoningProvider.check(lat, lon,
  object_type) -> ZoneVerdict`; класс-исключение для сетевых сбоев.
- `app/zoning/wfs.py` — `WfsZoningProvider`:
  1. Запрос `openmap:land_plots` по bbox вокруг (lat,lon). Нет участка/HTTP-ошибка/таймаут →
     переход к шагу 3 со `status=unknown`.
  2. Запрос водоохранной зоны (по региональному слою; регион определяем по городу объекта:
     Алматы→`almaty`, иначе пропускаем, если слоя нет). Попадание → `status=restricted`,
     `zone="водоохранная зона"`.
  3. Если участок есть и не ограничен → `status=allowed`, заполнить `land_use`/`kad_nomer`/площадь.
  4. **Сверка назначения с типом** (`object_type`): если `tsn_ru` явно исключает капитальное
     строительство данного типа (см. эвристику) → дописать предупреждение в `note` (вердикт при
     этом не меняем на restricted — это «обрати внимание», а не запрет).
- `app/zoning/heuristics.py` — сверка назначения↔тип:
  - Нормализуем `tsn_ru` в нижний регистр.
  - **Стоп-маркеры** для капитального здания (жилого/общественного/промышленного): назначение
    содержит `озеленен`, `благоустройств`, `сельскохозяйствен`, `для эксплуатации`/`обслуживания`
    (без `строительств`), `нестационарн`, `временн`, `парк`, `сквер`, `водн`. → предупреждение
    «целевое назначение участка («…») может не допускать капитальное строительство данного типа
    — уточните в местном акимате/ГАСК».
  - Иначе предупреждения нет. Эвристика консервативна (предупреждаем только при явном
    несоответствии), чтобы не плодить ложные тревоги.
- `app/zoning/__init__.py` — `get_zoning_provider()` (фабрика; даёт точку расширения и подмену в
  тестах).
- HTTP — через stdlib `urllib` с браузерным UA и таймаутом 8–10 с (без новых зависимостей; как
  `_check_link`/Satu). На таймаут/non-200/пустой ответ → `status=unknown`, `note` поясняет.

## Геометрия point-in-polygon
Малый bbox обычно возвращает 1 участок; при нескольких — выбираем содержащий точку **простым
ray-casting** по кольцам GeoJSON (без shapely; функция в `app/geo.py`:
`point_in_polygon(lon, lat, polygon) -> bool`, поддержка MultiPolygon). Если ни один не содержит
точку — берём ближайший по центроиду как «рядом», но `note` отмечает приблизительность.

## API (`routes.py`)
- `POST /api/objects/{id}/check-zone` → выполняет проверку, **сохраняет** вердикт в поля объекта,
  возвращает `ZoneVerdict`. 404 если объекта нет.
- `GET /api/zoning/wms?city=` → отдаёт конфиг слоя для наложения (URL `…/geoserver/ows`, список
  `layers` под город: `openmap:land_plots` + региональная водоохранная зона, `format=image/png`,
  `transparent=true`). Используется фронтом для `L.tileLayer.wms`.
- `GET /api/objects/{id}` (SP1) — расширить ответ полями `zone_*`, чтобы карточка показывала
  ранее сохранённый вердикт без нового запроса.

## Фронтенд (`app.js`, `styles.css`)
- На карточке объекта (`viewObject`) — блок **«Генплан/кадастр»** рядом с концептом:
  - Кнопка **«Проверить участок»** → `POST /check-zone` → бейдж вердикта:
    `разрешено` (ok), `ограничено` (danger/warn), `не проверено` (draft) — переиспользуем
    `statusBadge`-стиль. Под бейджем — целевое назначение, кад. номер, источник, при наличии —
    предупреждение о несоответствии.
  - Если у объекта уже есть сохранённый вердикт (`zone_status`) — показываем его сразу, кнопка
    становится «Перепроверить».
- **WMS-наложение:** в `baseLayers()`/на мини-карте объекта добавить overlay-слой
  `L.tileLayer.wms(wms.url, {layers, format:"image/png", transparent:true})` с тумблером в
  `L.control.layers` (как «Схема/Спутник», но это overlay). Конфиг — из `GET /api/zoning/wms`.
- Бейдж вердикта дублируем мелко в строке списка объектов (`#/objects`) — опционально, если
  `zone_status` есть.

## Обработка ошибок / мягкая деградация
- WFS недоступен / таймаут / не-200 / пустой ответ → `status=unknown`, `note`: «Геопортал не ответил
  — проверьте вручную», процесс не падает (как Satu/`_check_link`).
- Объект без координат — невозможно (lat/lon обязательны в SP1).
- WMS-слой не загрузился на карте — Leaflet просто не покажет тайлы; вердикт-проверка независима.

## Тестирование (SP2)
1. `point_in_polygon` — на известном квадрате: точка внутри → True, снаружи → False; MultiPolygon.
2. `heuristics` — «для благоустройства и озеленения» + «Жилой дом» → предупреждение; «для
   строительства жилого комплекса» + «Жилой дом» → без предупреждения.
3. `WfsZoningProvider.check` с **замоканным** HTTP (monkeypatch на функцию запроса; без сети):
   - участок есть, назначение под стройку → `allowed`, заполнены `land_use`/`kad_nomer`.
   - попадание в водоохранную зону → `restricted`.
   - HTTP-ошибка/пустой ответ → `unknown` (graceful).
   - участок «озеленение» + жилой дом → `allowed` + предупреждение в `note`.
4. `POST /api/objects/{id}/check-zone` (TestClient, провайдер замокан) → 200, вердикт сохранён в
   объекте; повторный `GET /api/objects/{id}` отдаёт `zone_*`. 404 для несуществующего объекта.
5. `GET /api/zoning/wms?city=Алматы` → конфиг с url и непустым `layers`.
6. Миграция-guard: на `building_objects` без `zone_*` колонок ALTER их добавляет (идемпотентно).
7. Регрессия: объекты/сметы SP1 работают как прежде; весь набор тестов зелёный.

**Сеть в тестах запрещена** (как в conftest): все WFS-вызовы мокаются. Живой запрос к
map.gov.kz — только ручной смоук при приёмке.
