# Подбор площадки и объекты строительства — дизайн

**Дата:** 2026-06-27
**Статус:** утверждён к планированию (детально — SP1; SP2/SP3 — контур, свои спеки позже)
**Связь:** расширяет Yale Building Calculator (ресурсная смета v0.3); смета привязывается к объекту.

## Видение

Пользователь нажимает кнопку → система **предлагает площадку под строительство** в выбранном
городе (Алматы/Астана), показывает её на карте, **сверяет с генпланом** (чтобы место не
попадало в зону планируемой застройки), и при согласии создаёт **карточку объекта**. На
карточке запускается создание и расчёт **сметы**, привязанной к объекту.

## Честная рамка (определяет всю архитектуру)

«Подбор со спутника, полностью автоматически» реализуется на **структурных данных OSM**, а не
анализом пикселей снимка (последнее — research-уровень CV, дорого и ненадёжно):

- **Свободный участок** — области, не покрытые footprint'ами зданий OSM (`building=*`) и не
  «built-up» landuse.
- **Размер** — площадь пустого полигона ≥ габарита здания (из сметы/ввода).
- **Инженерные сети** — близость к OSM-сетям (`power=*`, водопровод) либо прокси: близость к
  существующей городской ткани.

Спутниковые тайлы (Esri World Imagery) — **подложка карты для показа**, не источник детекции.
Это даёт настоящий автоподбор, инженерно-реализуемый геометрией над OSM-векторами (Overpass API).

## Декомпозиция (строим C→B→A: от твёрдого к рискованному)

| Подпроект | Что даёт | Сложность |
|---|---|---|
| **SP1 — Объекты + карта + привязка сметы** | Сущность объекта, карта (Leaflet), ручное создание объекта, «Создать смету» с объекта (FK) | 🟢 |
| **SP2 — Генплан: автопроверка зоны** | Адаптер зонирования на город, проверка точки/полигона на «планируемую застройку» | 🟡 (зависит от данных) |
| **SP3 — Автоподбор локации** | Кнопка «Предложить»: OSM-скан → кандидаты → кросс-чек генплана → топ-N на карте | 🔴 |

Каждый подпроект — самостоятельный рабочий инкремент со своим спеком и планом. **SP1 не
зависит от SP2/SP3**; SP3 переиспользует создание объекта (SP1) и проверку (SP2).

# SP1 — Объекты + карта + привязка сметы (детально)

## Модель данных

### Новая таблица `building_objects` (`models.py`)
```
id            int PK
name          str
city          str            # "Алматы" | "Астана"
lat           float          # центр объекта
lon           float
polygon       JSON           # GeoJSON-полигон контура участка (или None)
area_m2       float = 0.0    # площадь участка (по полигону/вводу)
status        str = "draft"  # draft | selected
source        str = "manual" # manual | auto (auto — из SP3)
score         float | None   # оценка пригодности (из SP3), None для ручных
notes         str = ""
created_at / updated_at  datetime
```

### Расширение `Estimate` (`models.py`)
```
object_id  int | None  FK -> building_objects.id   # смета привязана к объекту (0..N смет на объект)
```
Связь: `BuildingObject.estimates` (one-to-many), `Estimate.object` (many-to-one), nullable —
сметы без объекта продолжают работать как сейчас.

**Миграция:** `Base.metadata.create_all` создаёт новую таблицу автоматически, но **не добавляет
колонку `object_id` в существующую `estimates`** (ограничение SQLite create_all). Для dev это
решается пересозданием БД (gitignored, реседится при удалении — см. ybc-dev-setup). В коде
startup добавить идемпотентный guard: если в `PRAGMA table_info(estimates)` нет `object_id` —
выполнить `ALTER TABLE estimates ADD COLUMN object_id INTEGER`.

## Схемы (`schemas.py`)
```python
class ObjectCreate(BaseModel):
    name: str = ""
    city: str = "Алматы"
    lat: float
    lon: float
    polygon: Optional[dict] = None     # GeoJSON
    area_m2: float = 0.0
    notes: str = ""

class ObjectCard(BaseModel):
    id: int
    name: str
    city: str
    lat: float
    lon: float
    area_m2: float
    status: str
    source: str
    score: Optional[float]
    estimate_count: int
    updated_at: str
```

## Геометрия (без новых зависимостей в SP1)
Карта присылает полигон (GeoJSON, координаты lat/lon). Габариты и площадь считаем
приближением (плоская земля локально):
- `m_per_deg_lat = 111_320`; `m_per_deg_lon = 111_320 * cos(lat_центра)`.
- Bounding box полигона → `building_length`, `building_width` в метрах.
- `area_m2` — площадь по формуле шнурков в проекции метров (если полигон есть), иначе из bbox.

Модуль `app/geo.py`: `bbox_dims_m(polygon) -> (length_m, width_m)`, `polygon_area_m2(polygon) -> float`.
(shapely/pyproj НЕ нужны для SP1 — простая тригонометрия; тяжёлая геометрия придёт в SP3.)

## API (`routes.py`)
- `GET /api/objects` → `list[ObjectCard]` (сорт по updated_at).
- `POST /api/objects` (`ObjectCreate`) → создаёт объект; если `area_m2==0` и есть polygon —
  считает площадь; → `{id}`.
- `GET /api/objects/{id}` → объект + список его смет (id, name, status, total, version).
- `PATCH /api/objects/{id}` (name/notes/city) → `{ok}`.
- `DELETE /api/objects/{id}` → 204 (привязанные сметы НЕ удаляются, `object_id` обнуляется).
- `POST /api/objects/{id}/estimate` → создаёт смету-контейнер с `object_id`, **сидируя
  BuildingInput из объекта**: `city` = объект.city, `building_length/width` из bbox полигона,
  `project_name` = объект.name; статус draft; → `{estimate_id}`. Фронт переходит на смету.

## Фронтенд (`app.js`, `index.html`, `styles.css`)
- **Leaflet** через CDN (CSS+JS) в `index.html` (+ `leaflet-draw` для рисования прямоугольника/
  полигона). Вписывается в ванильный SPA без сборки.
- Навигация: добавить ссылку **«Объекты»** (`#/objects`).
- **Вид «Объекты»** (`#/objects`): карта (слои OSM-улицы + спутник Esri, переключатель) + список
  карточек объектов. На карте: поставить точку/нарисовать контур → панель «Создать объект»
  (имя, город) → `POST /api/objects` → карточка.
- **Карточка объекта** (`#/object/{id}`): мини-карта с контуром/точкой; поля (имя, город, площадь,
  заметки); список привязанных смет; кнопка **«Создать смету»** → `POST /objects/{id}/estimate`
  → переход на `#/estimate/{newId}` (существующая форма уже подхватит сиды).
- **Смета detail**: если у сметы есть `object_id` — показать бейдж/брейдкрамб «Объект: …» со
  ссылкой на карточку объекта.

## Обработка ошибок
- POST объекта без lat/lon → 422 (Pydantic).
- `/objects/{id}` несуществующий → 404.
- Полигон кривой/пустой → площадь 0, объект создаётся (площадь уточнит пользователь).
- Создание сметы для несуществующего объекта → 404.

## Тестирование (SP1)
1. `geo.bbox_dims_m` / `polygon_area_m2` на известном прямоугольнике → ожидаемые метры/площадь (допуск).
2. CRUD объекта: create → list → get → patch → delete (через TestClient).
3. `POST /objects/{id}/estimate` создаёт смету с `object_id`, city/габарит засеяны из объекта.
4. Удаление объекта обнуляет `object_id` у смет, сами сметы остаются.
5. Startup-guard добавляет колонку `object_id` на старой БД без неё (или create_all на свежей).
6. Сметы без объекта (`object_id is None`) работают как прежде (регрессия).

# SP2 — Генплан: автопроверка зоны (контур)

- **Адаптер `zoning`** на город: `ZoningProvider.check(city, point|polygon) -> ZoneVerdict
  {allowed: bool, zone: str, source: str, note: str}`. «Запрещено», если пересекает зону
  планируемой застройки/резерва генплана.
- **Спайк осуществимости (первая задача SP2):** отдают ли порталы машиночитаемый слой —
  [alag.kz](https://alag.kz/), [ggk.kz](https://ggk.kz/) (Алматы), [genplan.saulet.astana.kz](https://genplan.saulet.astana.kz/)
  (Астана) — через WMS/WFS/ArcGIS-REST. Есть → `WfsZoningProvider` (запрос по координате);
  нет → `OverlayZoningProvider` (наложить слой/тайлы на карту + подтверждение человеком).
- **Мягкая деградация** (как с Satu): нет данных/сбой → verdict `unknown` + ручное подтверждение,
  процесс не падает.
- API: `GET /api/zoning/check?city=&lat=&lon=` → ZoneVerdict; `GET /api/zoning/layers?city=` →
  ссылки/тайлы слоёв для наложения.
- Фронт: на карточке объекта и кандидате — вердикт генплана (бейдж allowed/blocked/unknown).

# SP3 — Автоподбор локации (контур)

- Бэкенд-зависимости: `shapely` (геометрия), Overpass-клиент (httpx/stdlib).
- **Скан:** для города/области (bbox) тянем OSM-объекты через **Overpass API** (тайлим область,
  кэшируем): здания, landuse, дороги, сети. Строим сетку-кандидатов (или используем пустые
  парцели), фильтр: (a) свободно (нет building footprints / built-up), (b) площадь ≥ требуемого
  габарита, (c) близость к сетям/городской ткани. **Кросс-чек каждого кандидата через SP2.**
  Скоринг (размер/доступ/сети/генплан) → топ-N.
- API: `POST /api/site-search` `{city, bbox, required_footprint_m2}` → `candidates[]`
  (полигон, площадь, score, факторы, вердикт генплана).
- Фронт: кнопка **«Предложить локацию»** → кандидаты-полигоны на карте → выбор → создаётся
  объект (SP1, `source=auto`, `score`).
- **Риски:** Overpass-лимиты на масштаб (тайлинг + кэш + ограничение области); данные сетей в
  OSM по КЗ редки (прокси + low-confidence); качество автоподбора — кандидаты суть предложения,
  человек подтверждает.

## Сводный стек и зависимости
- **SP1:** без новых backend-зависимостей (тригонометрия); фронт — Leaflet + leaflet-draw (CDN).
- **SP2:** HTTP-клиент к порталу (httpx/stdlib); парсер слоёв.
- **SP3:** `shapely` (+ возможно `pyproj`); Overpass-клиент. Добавляются в venv в SP3.
- БД: SQLite; новая таблица через create_all; `object_id` — startup-guard ALTER.

## Риски (сводно) и митигации
- Доступность слоёв генплана → спайк в SP2 + фолбэк на наложение/ручное подтверждение.
- Overpass масштаб/лимиты → тайлинг области, кэш, ограничение размера запроса (SP3).
- Редкие данные сетей в OSM (КЗ) → прокси по городской ткани + пометка low-confidence.
- Качество автоподбора → кандидаты = предложения; финальное решение и подтверждение генплана за человеком.
- Миграция `object_id` на старой dev-БД → идемпотентный ALTER-guard на старте.
